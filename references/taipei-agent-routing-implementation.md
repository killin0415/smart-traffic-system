# taipei-agent-routing — implementation notes

OpenSpec change `taipei-agent-routing` shifted the road dataset from
Kaohsiung VD to Taipei Section, and introduced the Gemini chat agent +
MCP routing tool that turns `chat.request` into structured route plans.
This note records the implementation choices that aren't already obvious
from the spec/design files in `openspec/changes/taipei-agent-routing/`.

## Data plane

- **Road network**: `data/taipei_road_sections.json`, generated by
  `scripts/import_tdx_road_network.py` (TDX Section + SectionShape
  endpoints, Taipei). Bbox 2.2 km around Taipei Main Station; 81 sections,
  97 nodes after dedup. Old `data/kaohsiung_road_sections.json` is retained
  on disk for history but no longer referenced.
- **VD removal**: `src/db/vd_sensor.py`, `tests/test_vd.py`,
  `infra/init-db/02-road-network-tables.sql`'s `vd_sensor` block, the
  `VDSensor` ORM model, and the `seed_vd_sensors` lifespan call were all
  hard-deleted. Existing DB was migrated by hand (`DROP TABLE IF EXISTS
  vd_sensor;`), and `traffic_node`/`traffic_edge`/`traffic_history` were
  truncated before the Taipei seed.
- **Live traffic**: `src/agents/traffic.py` returns to a Section-style
  fetch (`Live/City/Taipei`) but with the Taipei `-99` sentinel filter
  baked in (`_is_sentinel`). Rows whose `TravelSpeed`/`TravelTime` are
  ≤0 or whose `CongestionLevel` is `"-99"` are dropped before they reach
  Redis / TimescaleDB / graph weights.
- **Congestion factor**: spec D2 explicitly drops the dead-branch
  "speed≤0 → MAX" scenario; `_congestion_factor` now takes
  `float | None` and returns 1.0 for "no live data".

## MCP routing tool

`src/mcp_servers/routing_tool.py` exposes a single `plan_route` tool via
FastMCP. Pydantic models `PlanRouteInput` (with `extra="forbid"` so the
agent can't pass spurious args) and `RouteResponse` are the canonical
schema. `plan_optimal_route()` now returns `RouteResponse(...).model_dump()`
so the wire shape is single-sourced. The tool reads the in-memory graph
+ session factory off `kafka_runtime`, so tests can swap it in/out without
constructing the MCP transport.

## Gemini chat agent

`src/agents/chat_agent.py` builds a `genai.Client` per process. To capture
the routing tool's payload we wrap `plan_route` in a closure that records
successful results into a local dict; we then call `generate_content` with
`tools=[wrapper]` and let the SDK's automatic function calling handle the
back-and-forth. The wrapper-and-closure approach kept tests simple — they
mock `aio.models.generate_content` with a `side_effect` that invokes the
configured tool directly, then returns a fake reply.

Failure modes: missing `GEMINI_API_KEY` → `FALLBACK_REPLY`,
`asyncio.TimeoutError` (default 15s) → `GENERIC_ERROR_REPLY`, any other
exception → `GENERIC_ERROR_REPLY`. Service startup never depends on
Gemini being reachable.

## Kafka consumer extensibility

`TOPICS` is gone; `_resolve_topics()` reads `KAFKA_SUBSCRIBE_TOPICS`
(comma-separated, whitespace-trimmed; empty / whitespace-only falls
back to `chat.request,route.request`). `_consumer_loop` now logs and
skips on:

- Unknown topic (no handler registered) → WARN
- Malformed JSON or non-UTF-8 → ERROR
- Handler raised → ERROR

The thread keeps running in all three cases. `traffic.metrics` and
`handle_traffic_metrics` were removed entirely — graph weights are now
driven solely by TDX live polling.

## Tests

The change adds:

- `tests/test_traffic.py` — Section payload + sentinel filter (rewritten)
- `tests/test_routing_tool.py` — Pydantic schema validation, MCP tool registration, happy/sad paths
- `tests/test_chat_agent.py` — fallback / chitchat / route intent / tool error / timeout / API error
- `tests/test_consumer_extensibility.py` — env var parsing, unknown topic, bad JSON, handler crash

Total: **99 tests** (`uv run pytest -q`), up from 80 (after removing 17 VD tests and adding ~36 new ones).

## Phase 7 demo notes

Three throwaway demo scripts live under `backend/multiagent-service/scripts/`:

- `demo_taipei_live.py` — wires the runtime and runs one TDX live refresh; verified 685 sections fetched / 64 edges updated against the seeded 81-edge Taipei graph.
- `demo_plan_route.py` — calls `plan_route` directly. The original aspirational demo destination (信義誠品 ~25.0418, 121.5654) is **outside** the 2.2 km bbox set by spec D5, so the network has no path to it. The script instead uses Taipei Main → 善導寺 (~25.0444, 121.5238), which returns 2 in-graph routes (~0.74 km via 市民大道高架道路).
- `demo_kafka_chat.py` — produces a `chat.request`, runs one inline consumer iteration, publishes `chat.response`. Verified end-to-end with DeepSeek `deepseek-v4-flash` (see "LLM choice" below): the agent invoked `plan_route`, captured 2 routes, and DeepSeek's natural-language summary plus the full `route_payload` JSON were written to `chat.response` (Kafka correlation_id `e599c6e5`).

## LLM choice — switched from Gemini to DeepSeek (paid)

The original spec wrote "google-genai SDK". During Phase 7 demo we hit
serial roadblocks on the free-tier Gemini API key:

- `gemini-2.0-flash*` family: `RESOURCE_EXHAUSTED` with `limit: 0` (no
  free-tier quota at all on this account).
- `gemini-2.5-flash` and `gemini-2.5-flash-lite`: hallucinated tool errors
  even when `plan_route` returned a valid result, or returned an empty
  `response.text` after the tool call.
- `gemini-3-flash-preview`: 5 req/min and 20 req/day free-tier limits;
  also high latency that exceeded our default 15 s timeout.
- Gemma models (`gemma-3-*`): function calling is not enabled by the API.

User added a paid DeepSeek API key. We swapped the chat agent to
`AsyncOpenAI` pointed at `https://api.deepseek.com` (DeepSeek's API is
OpenAI-compatible). Default model is `deepseek-v4-flash` — the cheapest
tier — with `max_tokens=500` and `max_tool_loops=3` as per-turn cost
guards. Thinking mode is explicitly disabled (`extra_body={"thinking":
{"type": "disabled"}}`) to save output tokens and to avoid having to
round-trip `reasoning_content` between turns.

This is a deliberate spec deviation (the `agent-routing-tool` spec text
still says "google-genai SDK"). Behavior is identical from the
`chat.response` consumer's point of view: tool-calling pattern, captured
`route_payload` shape, and fallback on missing API key all match the
spec scenarios.

Env var changes: `GEMINI_API_KEY` / `GEMINI_MODEL` / `GEMINI_TIMEOUT_SECONDS`
are gone, replaced by `DEEPSEEK_API_KEY` / `DEEPSEEK_MODEL` /
`DEEPSEEK_BASE_URL` / `DEEPSEEK_TIMEOUT_SECONDS` / `DEEPSEEK_MAX_TOKENS` /
`DEEPSEEK_MAX_TOOL_LOOPS`. See `backend/multiagent-service/README.md`.
